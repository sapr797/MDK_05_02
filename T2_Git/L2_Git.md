Лекция 2.2. Система контроля версий Git. Организация работы в команде
Цель лекции:
Научиться сохранять код профессионально, работать в команде без страха всё сломать и использовать Git так, как это делают в реальных IT-компаниях.

Главное правило Git: каждое изменение — осознанное. Никаких "лишь бы работало" и безымянных файлов final_v3_исправлено_реально.zip.

Часть 1. Архитектура Git: как это работает внутри (за 10 минут)
1.1. Три состояния файла (запомните как таблицу умножения)
Любой файл в Git может находиться в одном из трёх состояний:

Состояние	Что значит	Команда для перехода
Modified (изменён)	Вы поправили код, но Git ещё об этом не знает	автоматически
Staged (подготовлен)	Вы сказали Git: "вот эти изменения хочу сохранить"	git add
Committed (зафиксирован)	Изменения навсегда сохранены в репозитории	git commit
Жизненный цикл файла (цифрами):
Рабочая папка (Modified) → git add → Staged → git commit → Репозиторий (Committed)

1.2. Три зоны Git (архитектура)
text
Рабочая папка          Область индекса          Репозиторий (.git)
(ваши файлы)     →     (Staging Area)      →    (история коммитов)
     ↓                       ↓                        ↓
  изменяете              git add                 git commit
  код                    подготовка              сохранение
Важно: папка .git — это и есть репозиторий. Она содержит всю историю, все ветки, все коммиты. Без неё Git не работает.

1.3. Что такое коммит простыми словами
Коммит — это снимок всех файлов в проекте на момент времени. У каждого коммита есть:

Уникальный хеш (например, a3f2b1c)

Автор и дата

Сообщение (что изменилось)

Ссылка на предыдущий коммит (родитель)

Представьте игру с сохранениями: вы можете загрузить любой момент и посмотреть, что было.

Часть 2. Базовые команды (15 минут практики)
2.1. Создание первого репозитория
Откройте терминал в VS Code (мы это уже умеем) и выполните:

bash
mkdir git_practice
cd git_practice
git init
Что произошло: создалась скрытая папка .git — теперь это репозиторий.

2.2. Первый коммит (пошагово)
bash
# Создаём файл
echo "# Мой проект" > README.md

# Смотрим статус
git status
# Вы увидите: Untracked files (файл не отслеживается)

# Добавляем в индекc
git add README.md

# Снова статус
git status
# Теперь: Changes to be committed (файл подготовлен)

# Сохраняем коммит
git commit -m "Первый коммит: добавлен README"

# Проверяем историю
git log
Важное правило сообщений к коммитам:

Начинаются с глагола: "Добавил", "Исправил", "Удалил"

Не длиннее 50 символов

На русском или английском — главное, чтобы понятно

Примеры:

✅ "Добавил функцию расчёта площади"

❌ "тут я поменял что-то"

2.3. Полезные команды на каждый день
Команда	Что делает	Когда использовать
git status	Показывает состояние файлов	В начале и конце каждой работы
git add .	Добавляет ВСЕ изменённые файлы	После того как сделали группу изменений
git add имя_файла	Добавляет конкретный файл	Когда хотите разбить изменения на несколько коммитов
git commit -m "текст"	Создаёт коммит	Когда подготовили всё нужное
git log --oneline	Краткая история коммитов	Быстро посмотреть, что было
git diff	Показывает неподготовленные изменения	Прежде чем сделать git add, проверить, что изменили
Часть 3. Ветки (Branches) — главная суперсила Git
3.1. Зачем нужны ветки
Без веток вся команда работала бы в одной папке — это хаос.
Ветки позволяют:

Разрабатывать новую функцию, не трогая рабочий код

Экспериментировать без страха

Вести несколько версий продукта одновременно

Представьте: основная ветка main — это "чистовик". А ветка feature/payment — черновик новой платёжной системы.

3.2. Команды для работы с ветками
bash
# Посмотреть все ветки (* — текущая)
git branch

# Создать новую ветку
git branch feature/calculator

# Переключиться на другую ветку
git checkout feature/calculator

# Создать и сразу переключиться (сокращённо)
git checkout -b feature/calculator
Практическое задание прямо сейчас:

bash
# Создаём новую ветку
git checkout -b feature/hello

# Создаём файл
echo 'print("Hello from feature!")' > hello.py

# Сохраняем изменения
git add hello.py
git commit -m "Добавлен hello.py в ветку feature"

# Переключаемся обратно на main
git checkout main

# Смотрим файлы — hello.py исчез! (потому что его нет в main)
ls
3.3. Слияние веток (merge)
Когда функция готова, её нужно вернуть в основную ветку.

bash
# Переключаемся в main (куда будем вливать)
git checkout main

# Вливаем изменения из feature/hello
git merge feature/hello

# Теперь hello.py появился и в main
Часть 4. Разрешение конфликтов (то, чего все боятся, но это просто)
4.1. Когда возникает конфликт
Конфликт — это когда две ветки изменили одну и ту же строку в одном файле. Git не знает, какой вариант правильный.

Создадим конфликт специально:

bash
# В ветке main
echo "version: main" > config.txt
git add config.txt
git commit -m "main: config"

# Создаём ветку feature и меняем ту же строку
git checkout -b feature
echo "version: feature" > config.txt
git commit -am "feature: config"

# Возвращаемся в main и меняем по-своему
git checkout main
echo "version: main UPDATED" > config.txt
git commit -am "main: config updated"

# Пытаемся слить — конфликт!
git merge feature
4.2. Как выглядит конфликт в файле
Git добавляет специальные метки:

python
<<<<<<< HEAD
version: main UPDATED
=======
version: feature
>>>>>>> feature
Всё между <<<<<<< HEAD и ======= — ваш вариант (текущая ветка).
Между ======= и >>>>>>> feature — вариант вливаемой ветки.

4.3. Пошаговое разрешение конфликта
Открыть конфликтный файл в VS Code.

Выбрать нужный вариант (или оба, или написать свой).

Удалить метки (<<<<<<<, =======, >>>>>>>).

Сохранить файл.

Добавить исправленный файл в индекс: git add config.txt.

Завершить слияние: git commit (откроется редактор с уже готовым сообщением — просто закрыть).

Готово! Конфликт разрешён.

Совет VS Code: нажмите Ctrl+Shift+P → Git: Merge Conflict → там есть кнопки "Accept Current", "Accept Incoming", "Accept Both". Удобно!

Часть 5. Стратегии ветвления. Gitflow
5.1. Простая стратегия (для 1-2 разработчиков)
text
main  ————●————●————●————●————●————
           \         /
feature/    ●———●———●
main — всегда рабочий код

feature/* — каждая новая функция или исправление

После завершения — вливаем обратно в main

5.2. Gitflow (стандарт для серьёзных проектов)
Gitflow — это как регламент: когда какую ветку создавать.

text
main       ————●————————————●————————————●————
              /            /            /
develop   ———●————●————●———●————●————●———
          /        \         \
feature/ ●——●——●    \         \
release/             ●————●    \
hotfix/                       ●——●
Типы веток (запомните):

Ветка	Назначение	Кто создаёт	Кто вливает
main	Только проверенный, рабочий код	автоматически	Tech Lead
develop	Для ежедневной разработки	автоматически	Любой разработчик
feature/*	Новая функция	Разработчик	Разработчик (в develop)
release/*	Подготовка к релизу	Tech Lead	Tech Lead (в main и develop)
hotfix/*	Срочное исправление ошибки	Tech Lead	Tech Lead (в main и develop)
5.3. Как работать по Gitflow (пример)
Начало новой функции:

bash
git checkout develop
git pull origin develop
git checkout -b feature/login-page
# ... работаем, делаем коммиты ...
git push origin feature/login-page
Функция готова (Pull Request):

bash
git checkout develop
git pull origin develop
git merge --no-ff feature/login-page
git push origin develop
git branch -d feature/login-page
--no-ff (no fast-forward) — сохраняет информацию о том, что это была отдельная ветка.

5.4. Что такое Pull Request (или Merge Request)
Это не команда Git, а процесс на платформе (GitHub, GitLab, Bitbucket):

Вы пушите ветку на сервер.

Открываете запрос на слияние.

Коллеги смотрят код (code review).

После одобрения — ветка сливается.

Для малых команд и учебных проектов: пока просто знайте, что это существует.

Часть 6. Организация работы в команде (реальные правила)
6.1. Золотые правила Git для команды
Никогда не коммитьте сломанный код. Каждый коммит на main или develop должен работать.

Коммитьте часто, пушите когда готово. Один коммит = одно логическое изменение. Не храните изменения неделями.

Всегда пишите осмысленные сообщения. Через месяц вы не вспомните, что значит "fix".

Перед созданием ветки обновляйте develop:
git checkout develop && git pull

Никогда не форсируйте пуш (git push --force) без крайней необходимости. Это перетирает историю коллег.

6.2. Типичный день разработчика в команде (скрипт)
bash
# Утро: синхронизация
git checkout develop
git pull origin develop

# Создаю ветку под задачу
git checkout -b feature/issue-42

# ... работаю, делаю коммиты ...
git add .
git commit -m "Добавил валидацию email"

# ... ещё работа ...
git add .
git commit -m "Написал тесты"

# Отправляю на сервер
git push origin feature/issue-42

# На GitHub создаю Pull Request, жду ревью

# После одобрения сливаю (через интерфейс GitHub)
